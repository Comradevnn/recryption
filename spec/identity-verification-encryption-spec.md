### 5.1 Identity verification

The design goal: verify the document and the person on-device, send the server only what it actually needs (a pass/fail result and a duplicate-detection value — not the document itself), and let orgs trust the outcome without ever handling anyone's ID. Below is the capture pipeline, then what leaves the device and why, then the specific algorithms for each job — "encryption," "hashing," and "signing" solve three different problems here and shouldn't be used interchangeably: encryption keeps data confidential, a signature proves who produced a result and that it hasn't been altered, and a hash/HMAC lets you check whether two values match without either side revealing the value itself. This section uses whichever of the three actually fits each step.

**Capture & extraction (on-device):**
1. Front of ID: camera capture, on-device OCR for name/DOB/photo/document type.
2. Back of ID, where applicable: decode the structured machine-readable data instead of OCRing free text — the AAMVA PDF417 barcode on US driver's licenses, or the MRZ (machine-readable zone) on passports and most national ID cards. Both are standardized, checksum-validated formats a barcode/MRZ reader decodes directly, far more reliable than OCR and much lighter-weight than a general document-AI pipeline.
3. **RESOLVED — a live selfie with a liveness check, face-matched against the ID photo, but only once, at initial verification.** This is scaled back from repeating it at every event, per the device-ticket design below (see "Event ticket," which replaces repeated face-matching with device-binding + a human visual check at check-in). It's kept once, at signup, because it's the only step that actually links "the person who owns this device" to "the person in the ID photo" — everything downstream (device biometric unlock, the hardware-bound key, the human check at the door) authenticates *that this is still the same device/person as signup*, not that signup itself was legitimate. Dropping this one-time check entirely is possible (see the tradeoff spelled out under "Event ticket" below) but isn't the default recommendation.
4. **RESOLVED — license, don't build.** Use a dedicated on-device ID-scanning SDK (Microblink's BlinkID or Regula) for MRZ/barcode extraction, licensing just the document-reading module rather than their bundled face-match add-on — since face-match is now needed only once (point 3), it's cheaper to call a standalone face-comparison API/library a single time per user than to license an entire always-on face-match module. Docling stays out of this pipeline entirely (see below) — it's the right tool for org-uploaded documents (501(c)(3) letters, waivers), not for MRZ/barcode/ID capture. Native OS frameworks (Apple Vision/VisionKit, Android ML Kit) remain the fallback for plain OCR if a lighter build is preferred over a licensed SDK.
5. **RESOLVED — out of scope for v1.** NFC chip reading (ICAO Doc 9303 passive authentication) is deferred; MRZ/barcode + one-time face-match + the human check at events (below) is the v1 bar. Revisit only if fraud data on passport verification specifically justifies the added engineering (an NFC read flow plus maintaining the ICAO Public Key Directory trust list).

**Local comparison, before anything leaves the device:**
6. Match the extracted name/DOB against what the user typed at profile setup — a plain equality/fuzzy-match check (allowing minor OCR noise or a missing middle name), done entirely on-device. No cryptography is needed for this specific step: both values already sit in the clear on the same phone, so there's nothing to protect until a result leaves it.
7. Confirm the extracted document type (available directly from the MRZ document-code field or barcode metadata) matches what the user claimed when starting verification.

**What actually reaches the server — and why it's a signature and a keyed hash, not "decryption":**
8. A **signed local attestation**: a small, timestamped payload (name match: true/false, DOB match: true/false, document type confirmed, expiry date) is what the server receives — not the document, not the raw extracted fields. It's authenticated with a digital signature, which proves what the device asserts and that it wasn't tampered with in transit; encryption on its own doesn't prove that, which is why "encrypt then have the server decrypt to check things match" isn't quite the right primitive for this step.
9. A **duplicate-document check**: alongside the attestation, send an HMAC-SHA256 of the document number, computed with a server-held secret key (a "pepper"), so the server can check "has this exact document already verified a different account" without ever seeing the plaintext number, and without the stored value being reversible even if the database is later exposed. This is precisely an HMAC with a secret key, not a bare hash: document numbers are short and structured enough in most countries that a plain hash could be brute-forced offline once the format is known, while a secret-keyed HMAC can't be recomputed without that server-side key.
10. That's the full list — no raw ID photo, selfie, or plaintext document number ever reaches the server. **RESOLVED — 90 days, then hard-deleted**, with one exception: if a dispute is actively open on a specific record, that record's hold extends until 30 days after the dispute resolves, logged and scoped to that record only — everyone else's 90-day clock is unaffected.

**Encryption at rest, for the retention window in #10 (envelope encryption — this is where actual encryption belongs in this pipeline):**
11. The ID photo, selfie, and extracted fields are bundled into one package on-device and encrypted with a freshly generated AES-256-GCM symmetric key (fast, authenticated, built for bulk data like images).
12. That one-time AES key is wrapped ("enveloped") using a public key only the verification service can open — the document is locked with a fast symmetric key, and that key is itself locked with an asymmetric key, so the bulk data never needs to be encrypted directly with the (slower) asymmetric algorithm.
13. The private key that unwraps the envelope lives in a Hardware Security Module or cloud KMS (AWS KMS / Google Cloud KMS or equivalent), never in application code or a plain database.
14. On-device, the raw ID sits behind the phone's own secure hardware (iOS Keychain/Secure Enclave, Android Keystore), requiring a biometric or passcode unlock, not just ordinary app storage.
15. Face photos are biometric data and should get their own explicit consent screen, separate from the general Terms of Service (Illinois' BIPA and GDPR's special-category-data rules both treat biometric data more strictly than ordinary account info).

**Event ticket — proving verification to the org, without the org re-scanning anything, built around the device+barcode idea:**

Your instinct here was right, and it maps directly onto the signed-credential concept above — a "ticket" *is* the signed badge, rendered as something scannable. Working through it precisely:

16. **RESOLVED — a fresh ticket is issued per confirmed shift, not on a recurring schedule.** Since a user already gets matched to a specific shift before this matters, generate the signed credential at the moment a pairing is confirmed, scoped to that shift's date/venue, rather than reissuing on a generic weekly cadence. This resolves the earlier open refresh-interval question directly: the ticket is only ever valid for the event it was issued for, so there's no separate revocation-latency window to reason about — an old ticket simply doesn't match the current shift.
17. The ticket contains: name, **verified ID photo** (not a raw scan of the document — same minimization principle as the rest of this section, just the photo plus a verified checkmark), verification status, shift ID, and venue/date — all signed with the platform's private signing key, exactly as before.
18. **On "cached version of their ID" — worth narrowing this to a cached badge, not a cached document.** Displaying the actual ID scan in-app would reopen the exact problem the rest of this section avoids (name, document number, address, etc. now visible to whoever's at the check-in table). What should be cached and shown is the same minimized badge from point 17 — photo and verification status, nothing else from the document.
19. **Device-binding, so a screenshot or forwarded ticket doesn't work on another phone:** at check-in, alongside scanning the ticket's QR/barcode (a pure signature check, offline, same as before), the org's scan can present a short challenge that the presenting device signs using its own hardware-resident key (the P-256 key from the table below) — proving that *this specific device*, the one that completed verification, is physically present, not just that someone has a copy of the ticket's image. This is the concrete answer to "as long as the device is the same that is brought to the event."
20. **Device biometric unlock is what "verification using biometrics" actually buys you here** — Face ID/fingerprint/passcode gates unlocking the phone (and, underneath it, the hardware key) in the first place, so whoever is holding and unlocking the phone to show the ticket has already passed the device's own biometric check. That's a genuinely useful signal, but it's worth being precise about what it proves: it confirms "the person unlocking this phone is whoever enrolled biometrics on it," not "that person matches the ID photo." Those are two different claims, and only the one-time face-match (point 3) links the second one — device biometrics carry the trust forward from that point, they don't establish it independently.
21. **At the door, a human does the last check — the same way ID checks work everywhere else already:** the org staff member compares the volunteer's physical ID, the person standing in front of them, and the photo shown on the ticket. Three-way visual comparison, no algorithm required, which is exactly why this can replace repeated automated face-matching rather than needing to reproduce it in software.

**The one honest gap, worth deciding explicitly rather than leaving implicit:** none of points 19–21 verify that the person who *originally* completed device setup is the person in the ID photo — that link only exists if something checked it once, which is what point 3's one-time face-match is for. Two real options:
- **Default recommendation: keep the one-time face-match at initial verification.** It's a single API call per new user (not a per-event cost), and it's the only thing in this whole design that actually connects "this device" to "this document's photo" in the first place.
- **Fully matches what you described: drop it entirely.** Verification then rests on the document's own photo being accurate, checked for the first time by whichever human process exists at someone's *first* event (or not checked by a human at all, if first verification happens with no staff present). Simpler to build, and reasonable given the human check now baked into every event regardless — but worth choosing this on purpose rather than by default, since it's a real reduction in what "verified" is actually attesting to.

**Algorithm choices — there's no single "best" one, since each job below is a different cryptographic problem:**

| Job | Algorithm | Why |
|---|---|---|
| Encrypting the retained package (photo/selfie) at rest | **RESOLVED:** AES-256-GCM | Authenticated, fast, hardware-accelerated on essentially every modern phone |
| Wrapping that AES key for the verification service ("envelope") | **RESOLVED:** X25519 (ECDH over Curve25519) | Modern, fast, small keys; only fall back to RSA-4096 if a specific vendor or compliance requirement forces it — no such requirement currently identified |
| Signing the issued verification credential/badge | **RESOLVED:** Ed25519 | Fast, deterministic (no nonce-reuse footguns like ECDSA), the current default for exactly this job — it's what orgs verify offline |
| Hardware-resident device signing key (the key that never leaves the phone's secure hardware) | **RESOLVED:** P-256 (secp256r1) | Not really a free choice — this is what Apple's Secure Enclave and most Android Keystore/StrongBox implementations actually support for hardware-bound keys; Curve25519 keys can't currently be generated inside that hardware boundary on iOS |
| Device proof-of-possession at check-in (event-ticket point 19) | **RESOLVED:** same P-256 key, used to sign the org's check-in challenge | No new key needed — this is a second use of the existing hardware-resident device key, not a separate algorithm decision |
| Duplicate-document check | **RESOLVED:** HMAC-SHA256 with a server-held secret pepper | A bare hash of a short, structured field like a document number is guessable via brute force once the format is known; a secret-keyed HMAC isn't |
| Everything in transit | **RESOLVED:** TLS 1.3 | Baseline, worth stating explicitly rather than assuming |

No optional paid background-check tier at launch — ID verification plus the signed badge is treated as sufficient trust signal, with plain language to organizations making clear that verification confirms identity, not suitability, and that the decision to work with any given volunteer remains the org's own.

## Touchpoints elsewhere in the master doc (for context when merging)

These aren't part of the crypto architecture itself, but they're where it plugs into the rest of the product — worth keeping in view so the merge doesn't lose the connective tissue:

- **Onboarding (3.1, step 6):** "Identity verification — selfie-to-ID match via a verification vendor (see 5.1). Required before a user's first pairing is confirmed, not before browsing." This is what triggers the whole pipeline above, and confirms it's gated on first pairing, not on signup itself.
- **Pairing reveal (3.2):** each user's brief partner profile includes the **verification badge** alongside name/age/occupation/intro/history — this is the signed credential from step 16 surfaced in-app, not a separate display mechanism.
- **Premium filters (1.6):** premium's advanced filters include a "verified-badge tier," so the badge's verification-status field needs to be filterable, not just a boolean shown on a profile.
- **Org-side check-in (org-side flow section):** "A simple check-in tool for shift day (so 'did they actually show up' and 'did they actually contribute' are org-verified, not just self-reported)" — this is the same check-in moment where the org's device scans the credential (step 17); the attendance-tracking tool and the verification-scan tool are the same touchpoint, not two separate features.
- **Constraint (section 7):** "Never require background-check-tier verification to use the core product, only to unlock the optional higher-trust tier" — a hard constraint on this system: the pipeline above is the baseline, not a paid add-on tier.
- **Org verification is a separate, non-cryptographic concern:** the org-side flow also mentions verifying that an organization is a registered 501(c)(3) before it can list shifts. That's a business/legal verification (checking nonprofit status against a registry), unrelated to the person-identity/encryption pipeline above — worth not conflating the two when merging, since they'll likely live in different parts of the org-onboarding spec.
